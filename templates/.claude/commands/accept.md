你是编排器。目标 spec：`$ARGUMENTS`。这是人与团队的**验收闭环**，必须闭合，不能停在"等人再说"。

## A. 出验收材料
1. 汇总：每条 AC 的测试结果（✅/❌ + 用例名 + 证据）、覆盖率、评审结论、与原型对齐情况。
2. 生成/更新 `docs/reviews/$ARGUMENTS-验收.md`：AC 逐条核对表、演示方式（如何启动 / 账号 / 数据）、风险与已知问题、待人判断的业务点。
3. **spec 同步确认（见 CLAUDE.md §4.5）**：若本次改动涉及 spec（逻辑/行为变、或新增回归 AC），**把 spec 的 diff 一并列出**让人随验收一起拍板；纯 bug/重构则注明"无需改 spec"+ 原因。
4. 给结论建议（可验收 / 待修复 + 阻断项），然后**等待人的判定**。

## B. 按人的判定闭环（关键，别再追问"要不要修"）
- **人说「通过 / 验收 / 没问题」** → 收尾：
  - 若有待确认的 **spec diff → 合并进 `docs/specs/$ARGUMENTS*.md`**；该文件 status 置 `accepted`；`docs/.state/$ARGUMENTS.json` 的 `accept` 置 `pass`；跑 `node tools/board.mjs`；告知已验收。

- **人提出任何问题 / 不满意（哪怕一句话）** → **不置 accepted，立即自动驱动修复**：
  1. 在 `docs/changes/CHG-<日期>-<简述>.md` 记录问题清单，关联受影响 AC；`docs/.state/$ARGUMENTS.json` 相关节点置 `doing`，跑 board。
  2. `Agent(dev)` 接手：若涉及需求变化先改 spec 的 AC → 针对每个问题**先写能复现的回归用例（红）→ 修代码使其转绿（后端/前端一并）→ 编译 + 跑测试 + 连真库冒烟**。
  3. 全绿 → 更新交付说明（写清修了哪些 + 真库冒烟证据），`docs/.state/$ARGUMENTS.json` 的 `accept` 置 `wait`，跑 board（控制台 🔔 提醒你**复验**）。
  4. 回到 A，**循环直到你明确「通过」**。

全程除"等你给验收判定"外不打扰你；**用真子代理 `Agent(dev)` 干活，编排器自己不改业务码**；禁止改测试迁就错误实现。
