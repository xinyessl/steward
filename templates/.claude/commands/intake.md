用户会给一个**飞书文档链接**（里面是一堆 bug / 需求）。按下面走，别跳步：

## 1. 拉正文
运行：`node tools/feishu-fetch.mjs "<飞书链接>"`
- 成功 → 拿到文档纯文本，进入分诊。
- 报错（没配机器人 / 无法确定项目 / 权限 / 旧版文档…）→ **把原话转告用户**，让其在控制台顶栏「飞书配置」补齐或换新版文档链接，然后**停下**。

## 2. 分诊（按 CLAUDE.md §4.6，对照历史 spec）
把正文拆成 N 条独立项；**逐条**判定去向：
- **改系统行为 / 是需求** → 对照 `docs/specs/*`：命中现有 spec 就标「改 SPEC-XX 的 AC-几」，否则「新建 draft」；并标**影响面**（depends_on、共享表/字段、共享接口、同模块）。
- **纯零碎、不改规约** → 标「→ 速记」。
- **BUG**：暴露了 spec 漏洞/边界 → 往 spec 补一条回归 AC；纯代码修(spec 没错) → 「→ 速记」或直接 `/fix`。

## 3. 出「分流清单」给用户确认（硬门）
一张表：**每条 → 建议去向 + 理由 + 影响面**。用户可逐条改去向。**不确认不落地。**

## 4. 确认后落地
- spec：新建 draft / 更新现有（给 diff）。
- 速记：用 `/todo` 那套（node 安全追加进 `docs/tasks.json` 的 `items`，shape: `{id,text,done:false,pri,ts}`）。
- **铁律：一条只去一个地方，spec 与速记绝不并存同一件事。**

## 5. 回报
一句话：建/改了哪些 spec、记了几条速记、有没有需要你进一步定的。
