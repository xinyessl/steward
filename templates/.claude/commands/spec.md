你是编排器。把「$ARGUMENTS」转成/更新为可验证 spec。**$ARGUMENTS 可能含多条需求——按 CLAUDE.md §4.6 先拆条、算影响面、给人确认，再产出 spec。不写代码。**

步骤：
1. **判断阶段 + 拆需求**：
   - **绿地（还没代码 / 刚 `/init` 完）→ PRD（`docs/PRD.md`）+ 原型图（`docs/prototype/`）= 事实源**，照它们拆。
   - **存量（已有代码）→ 以代码现状 + 现有 spec 为准**；项目里的 PRD/原型可能已过时、仅参考。
   - 两种都先拆成 N 条独立需求列出（每条一句话 + 归属功能模块），来源含 `$ARGUMENTS`。
2. **影响面分析（§4.6）**：每条 + 跨条，顺 `depends_on` / 共享数据表字段 / 共享接口 / 同模块 扫出**受影响的已有 spec**，并标多需求之间的冲突点 → 产出「**影响面清单**」给人确认。
3. `Agent(dev)` 按 `docs/specs/_TEMPLATE.md`，在 `docs/specs/` **新增/更新所有相关 spec**（含可自动化判定的 AC、接口契约、数据契约、测试要点、DoD）；`module:` 填所属功能模块。**存量有库时：数据字段对照「①本项目信息 → 真实库结构文件」逐字段核对，禁止臆造列名，查不到 → `NEEDS-HUMAN`**；**绿地无库时：spec 的数据契约 = 新表/字段设计，在 spec 里写清、随影响面清单请人确认（这是设计、不是臆造）**。
4. 汇总：各 spec 路径、AC 数、依赖、**影响面清单**、所有 `NEEDS-HUMAN` 未决业务点。
5. status 置 `draft`，提示评审通过改 `ready`，再 `/build <id>` 开发。

模糊业务点一律 `NEEDS-HUMAN` 列出，禁止臆造。
