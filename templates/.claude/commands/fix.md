你是编排器。问题/需求：「$ARGUMENTS」。**精简版 spec-first 闭环。$ARGUMENTS 可能含多条——先按 CLAUDE.md §4.6 拆条 + 影响面分析，再让开发 agent 一并干完：**

0. **拆条 + 影响面分析（§4.6）**：把 $ARGUMENTS 拆成 N 条独立改动；逐条 + 跨条，顺 `depends_on` / 共享数据表字段 / 共享接口 / 同模块 扫出**所有受影响的 spec**，标多条之间的冲突点 → 出「影响面清单」（涉及多文件/模块/不确定时先给人确认）。
1. **定位 + 记录**：每条归属到已有哪条 spec（新需求则现写）；在 `docs/changes/CHG-<日期>-<简述>.md` 建变更单，关联**全部**受影响 spec 与 AC。
2. `Agent(dev)` 接手：**开工前读两层经验库避坑（`docs/lessons.md` + `$STEWARD_LESSONS`，§4.7）→ 先写能复现缺陷的回归用例（红）→ 改码使其转绿（后端/前端一并）→ 编译 + 跑测试 + 连真库冒烟**。写持久层前必读「①本项目信息 → 真实库结构文件」，禁止臆造列名。
3. **分类并处理 spec（见 CLAUDE.md §4.5）**：
   - 逻辑/行为变，或缺陷暴露了 spec 漏的边界 → **起草 spec 的 diff**（改哪条 AC / 新增回归 AC），随复验材料附上、待你确认。
   - 纯 bug 修复 / 重构（行为不变）→ **不动 spec**，CHG 里记一行原因即可。
4. **沉淀经验（§4.7）**：本次若踩了非显然的坑 → 追加一条教训 + 自检项，**分流**：项目专属→`docs/lessons.md`，通用→`$STEWARD_LESSONS`。fix 是经验最该沉淀的入口。
5. **交你复验**：把对应 spec 的 `docs/.state/<id>.json` 的 `accept` 置 `wait`（控制台 🔔），跑 `node tools/board.mjs`；输出变更摘要 + 真库冒烟证据 +（若涉及）**spec diff** +（若有）新增教训。

禁止改测试迁就错误实现；**用真子代理 `Agent(dev)`，编排器自己不改业务码**。spec 的最终改动以你复验时确认的 diff 为准。
