> 流程 spec = 一条**端到端、跨模块**的业务流程怎么跑通。流程驱动的系统（审方/开单/审批/结算/分诊…）以它为骨架，
> 功能 spec 挂在各步骤上。核心价值：跨环节的问题（"药师没拿到任务"）能一眼定位到流程里某一环（任务推送的 WebSocket 依赖）。

## 1. 触发 / 角色 / 前置
- **触发**：什么事件/操作/定时启动这条流程。
- **参与角色**：<角色1>、<角色2>、<系统/服务>…
- **前置条件**：进入本流程前必须成立的状态。

## 2. 主流程（时序 / 泳道）
按步骤编号。每步写清：**谁 → 做什么 → 调哪个接口/服务 → 改哪个状态 → 依赖什么（端口 / WebSocket / MQ / 定时任务 / 外部系统）**。
1. …
2. …
（可加 mermaid `sequenceDiagram` 时序图；步骤多时优先编号步骤表，一行一步。）

## 3. 状态机（核心对象状态流转）
核心业务对象（如 处方 / 审方单 / 工单）：**当前状态 —(触发事件/条件)→ 目标状态**（+ 副作用）。
- <状态A> —<事件>→ <状态B>
- …
（可加 mermaid `stateDiagram-v2`。）

## 4. 异步 / 推送 / 定时环节（排障最常卡这一段，务必写全）
逐个列本流程里非同步请求-响应的环节：
| 环节 | 机制 | 走哪个服务（地址·端口·协议） | 触发/订阅 | 失败表现 |
|---|---|---|---|---|
| 任务推送 | WebSocket | 如 `http://192.168.1.192:8893` ws | 药师上线订阅 / 新任务广播 | 端口未放行→连不上→收不到任务 |
| … | 轮询 / MQ / 定时 | … | … | … |
> ⚠ **端口 / 地址 / 中间件这类基础设施依赖必须写死清楚**——它们不在业务代码里、最容易被漏，却是现场故障高发点。

## 5. 跨边界与依赖
本流程横跨的 模块 / 服务 / 库 / 外部系统；每个标注 **硬依赖（挂了流程就断）** 还是 **软依赖（降级可继续）**。

## 6. 异常 / 超时 / 兜底
关键分支：超时怎么处理（如审方倒计时到点）、失败重试、降级路径、人工兜底。

## 7. 关键失败点 · 排障 checklist（高价值，把踩过的坑固化）
「哪一环挂了 → 什么现象 → 怎么定位」，一行一条：
- <环节> 依赖 <X（端口/服务/中间件）> 不可用 → 现象：<用户可见症状> → 排查：<查什么>
- 例：任务推送依赖 8893(WebSocket) → 现象：药师在线但收不到新任务 → 排查：8893 是否放行 / WS 是否连上 / 中间网络设备
- …

本流程各步骤落在哪些功能 spec 上（id 列表，双向回链）：<FEAT-id>（第N步）…
