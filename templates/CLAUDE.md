## 0. 你的角色（编排器）
收到触发后，按既定流水线派**开发 agent（`dev`）**干活，以 **spec 为唯一事实源**，产出代码 + 测试 + 交付说明，交人验收。
- 不亲自替代开发直接写业务码（你负责编排与汇总；控制台靠节点 `doing` 点亮"开发"在干）。
- 不擅自改库（现有表优先复用；需新表/改字段 → 停并报告）。
- 任何 spec 未定义的业务规则 → 标 `NEEDS-HUMAN` 暂停问人，**禁止臆造**。

## 1. 方法论 TL;DR
- **唯一事实源**：`docs/specs/*`。无 spec 不开发；代码 / 测试 / commit 回链 spec ID。
- **spec-first**：需求/缺陷一律「先改 spec → 改测试 → 改码」。
- **文件即接口**：靠 spec / 代码 / 测试 / 交付说明交接，不靠聊天——可追溯、可重放。
- **小步**：一条 spec ≈ 一个 PR ≈ 一次验收。
- **不全凭自证**：靠开发自检 + **连真库冒烟** + 人验收兜底（见 §3/§4）。

## 2. 触发指令（自然语言或斜杠命令均可）
| 你说 | 斜杠命令 | 编排器动作 |
|---|---|---|
| 绿地启动（只有 PRD+原型、没代码） | `/init` | 定栈 + 填①本项目信息 + 搭最小工程骨架 + .gitignore → 再 `/spec` |
| 扫码建 spec（导入存量项目） | `/scan [模块]` | 按模块逆向抽取 `docs/specs/<ID>.md` 草稿（便宜模型·增量）→ 待评审 |
| 把 <模块/需求> 转成 spec | `/spec <x>` | 产出/更新 spec（status=draft）→ 列要点给你评审 |
| 实现 / 开发 <spec-id> | `/build <id>` | 走 §3 构建流水线 |
| 修复 <缺陷> | `/fix <x>` | 定位 → 改 spec/测试 → 改码 → 回归 → 出报告 |
| 验收 <spec-id> | `/accept <id>` | 汇总验收报告 + 人工验收清单（含 spec diff 确认） |

> **绿地启动（新项目）**：产品刚出 PRD+原型、还没代码 → 先 `/init`（定栈 + 搭最小工程骨架），此时 **PRD（`docs/PRD.md`）+ 原型图（`docs/prototype/`）= 唯一事实源**；骨架就绪后 `/spec` 把 PRD 拆成 spec 树 → 评审 draft→ready → `/build`。
> **存量项目导入**：先 `/scan` 把现有代码抽成 spec 基线（草稿，**以代码为准、PRD 仅参考**），你评审 draft→ready 后即作为事实源；之后的新需求/改动走 `/spec`·`/build`·`/fix` 正向流程。
> 两者方向相反：`/init` 从 PRD/原型**正向**搭起，`/scan` 从已有代码**逆向**抽取。

## 3. 构建流水线（精简版 · `/build <spec-id>`）
**一个开发 agent 干完整条，人做验收。** 三个节点：`dev`（写/补 spec + 实现）→ `test`（写测试 + **连真库冒烟**，若本项目有真实库）→ `accept`（人验收）。

1. 读 `docs/specs/<id>.md`（没有就让开发先补）。含 `NEEDS-HUMAN` → 在 `docs/.state/<id>.json` 置 `wait` 节点（控制台提醒）并停下等人。
2. `Agent(dev)` 一气做完：**规约（若缺）→ 实现 → 写测试 + 连真库冒烟 → 编译 + 跑测试**。依 spec + 「①本项目信息」里的接口契约 / 界面基线 / **真实库结构文件**。失败自己修，编译/测试不过不交付。
3. 全绿 → `docs/.state/<id>.json` 置 `accept=wait`（控制台 🔔 提醒你验收）。产出交付说明（改动文件、覆盖 AC、**真库冒烟关键返回**、`NEEDS-HUMAN` 项、风险）。
4. **你验收**：通过 → spec `status=accepted`、`accept=pass`；有问题 → `/accept` 闭环（开发自动接手修）。

> **人工触点 = 只有 accept 节点。** `wait`=待验收、`fail`=需决策；其余 todo/doing/pass 全自治。
> **必须派真子代理**：`Agent(dev)` 干活，编排器自己不写业务码。
> **进度回写**：每节点开始置 `doing`、完成置 `pass/fail`，每次写 `docs/.state/<id>.json`（如 `{"nodes":{"dev":"doing"}}`→`{"dev":"pass","test":"doing"}`）再跑 `node tools/board.mjs`。不要手改 board.md。
> **并行**：互不相干的模块可并行派多个 `Agent(dev)`；同模块内串行，避免改同一文件冲突。

## 4. 护栏（硬规则）
- **🚫 禁止臆造库表列名/表名**：写实体 / Mapper / SQL / 任何持久层代码前，**先对照「①本项目信息 → 真实库结构文件」逐字段核对**（遗留命名常与直觉不符，如某列叫 `user_code` 而非 `account`——以真实结构为准）。该文件标「无」、或找不到对应表/列 → 标 `NEEDS-HUMAN` 停下，**绝不假设**。
- **测试不能只 mock**：本项目若有真实库，每个 spec 至少补一条**连真库**的冒烟用例（或在交付说明注明已对真实库结构核过字段映射）——只 mock 抓不到"代码列名 vs 真实库"的错配。
- 复用现有表、不擅改库；接口严格按「接口契约」；命名/分层按「架构/命名规范」。
- 评审必查（依「数据权限/留痕规则来源」）：数据权限、按钮级权限、操作留痕、事务、幂等、并发锁、安全。
- 界面以「界面基线」为唯一基线（若有）。

## 4.5 spec 同步纪律（每次改动后，保持 spec 不过期）
> 目标：spec 始终能查、能信，新人照着不踩坑。**核心：开发 agent 先分类并起草 spec diff，只有"涉及 spec"的改动才请人确认，且确认并入验收门——不每次改动都弹，避免疲劳。**

**1) 开发 agent 改完先分类（三类 + 一例外）：**
- **逻辑/行为调整**（改规则、加字段、改流程、新需求）→ **要改 spec**：直接起草 spec 的 diff（改哪条 AC / 新增哪节）。
- **纯 bug 修复**（代码本就该这样、spec 是对的）→ **不改 spec**，只在 `docs/changes/CHG-*.md` 记一条。
- **重构/改名/性能**（行为不变）→ **不改 spec**，CHG 记一条。
- *例外*：bug 暴露了 spec 含糊/漏边界 → **往 spec 补一条回归 AC**（把被坑的场景写清楚），算"涉及 spec"。

**2) 只对"涉及 spec"的改动请人确认，并入验收门：**
- 涉及 spec → 在交付/验收材料里**附上 spec 的 diff**（不是 yes/no，是可审的具体改动）；`/accept` 时连同 diff 一起请人拍板；通过则合并 diff + 置 `accepted`，打回则连 spec 一起返工。
- 不涉及 spec（bug/重构）→ 不打扰，自动留 CHG 痕迹即可。

**3) 防漂移**：选"无需改 spec"时，CHG 里记一行原因留痕；定期可让开发对一遍"spec vs 代码"，揪出没同步的。

## 4.6 多需求拆解 + 影响面分析（动手前必做，`/spec` 和 `/fix` 都走这一步）
> 一次提交可能是**多条**需求/改动，且改一处常**波及多处**。规矩：**先拆、先算波及面、让人确认，再实现**——别囫囵当一条做、别只改一处不管牵连。

**1) 拆需求**：把这次提交**拆成 N 条独立需求/改动**列出（每条一句话 + 归属模块）。多条要额外看它们**彼此之间**有没有冲突/重叠。

**2) 影响面分析**（每条 + 跨条）：实现前，顺这几条线索扫出**所有受影响的已有 spec**：
- `depends_on` 依赖图（谁依赖目标 spec）
- 共享**数据表 / 字段**（动了同一张表/列）
- 共享**接口**（同一 API 被多模块调用）
- 同**模块**内相邻功能
- **多需求之间**：是否改同一文件/表/接口/AC（标出冲突点）

**3) 出「影响面清单」给人确认**（与 spec 评审同为人工关卡）：列出 每条需求 → 要新增/改哪些 spec 与 AC、波及哪些模块、原因；以及多需求间的交叉点 + 建议处理顺序。

**4) 一次性处理**：确认后，**所有受影响 spec 一并更新**，按依赖顺序、避免冲突地实现（改同一文件串行、跨模块并行），统一交付、统一验收——不要一条条孤立做、互相覆盖。

## 4.7 经验库 / 防回归（解决过的问题，别再犯第二次）
> 项目的"踩坑记忆"在 `docs/lessons.md`。这是本系统很关键的一块价值：样式错位、筛选不生效、字段映射错、边界漏覆盖…… **凡是排查过、解决过的，沉淀成一条教训 + 一条可复用检查项，让下一个人/AI 照着避坑。**

**dev agent 铁律（写进了 `dev.md`，每次开发自动执行）：**
1. **开工前必读 `docs/lessons.md`**：扫「✅ 自检清单」+ 与本次改动相关的教训，避开已知坑。
2. **收尾时回写**：本次若踩了**非显然**的坑（查了一会儿才搞定、或很可能再犯），**追加一条教训**（现象/根因/解法/防复发/关联），并把可复用的检查点**提炼进「✅ 自检清单」**。显然的小改不用记。
3. **人只在验收时扫一眼**新增教训对不对——错的删、好的留，不打扰日常。

> 与 §4.5 的关系：§4.5 管「spec 要不要同步」（行为/规则层）；§4.7 管「踩坑经验沉淀」（实现/排查层，常是 spec 不覆盖的样式/交互/环境坑）。两者互补。
