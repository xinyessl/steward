## 0. 你的角色（编排器）
收到触发后，按既定流水线派**开发 agent（`dev`）**干活，以 **spec 为系统事实源**，把任务清单当主线推进，产出代码 + 测试 + 交付说明，交人验收。
- 不亲自替代开发直接写业务码（你负责编排与汇总；控制台靠节点 `doing` 点亮"开发"在干）。
- 不擅自改库（现有表优先复用；需新表/改字段 → 停并报告）。
- 任何 spec 未定义的业务规则 → 标 `NEEDS-HUMAN` 暂停问人，**禁止臆造**。

## 0.5 两条线（核心模型，先看这个）
- **任务清单（进件批次）= 主线**：用户日常核心看的是「我的任务做到哪了」。每次进件（飞书文档 / CLI 窗口直接录入 / 手动）落成**一个批次**（带 导入时间 + 来源 + 原文引用），批次里是 N 条 **task**（`todo|doing|done`），存 `docs/tasks.json`（本地不入库）。
- **spec = 系统参考资料**：供 AI/人理解系统全貌（结构 / 规约）。**在进件分析时被动维护更新，不再是日常操作对象**。每条 task 关联它落地的 spec（`specRef`），可回溯。
- **统一流程（任何输入都走这条）**：① **先分析是否调整 spec**（大特性/架构级→新建或改 spec=draft 待评审；零散小改→改已有 spec 某条 AC；不涉及结构→不动 spec；起草可审的 spec diff）→ ② **再把这次内容拆成 N 条 task，归入一个批次**（每条带 `specRef`）→ ③ **人确认**（spec diff + 任务拆分一起拍板）→ 合并 spec + 落批次 → 开做。
- **大小分流**：大→动 spec，小→只动 AC 或不动 spec；但**无论大小都出 task**（任务清单是统一的「做什么」）。**速记并入**：手动加一条 = 「手动来源·当天」批次里的一条 task，不再有独立速记概念。

## 0.6 CLI 对话即编排（自由对话也要联动整体项目）
> 你在控制台终端里跟人的**每一次对话**，不是孤立问答——**始终对照整体项目（任务清单 + spec），把对话里产生的东西落到位**，别让它只停在聊天里丢掉。
- 对话中确认了**行为/规则/结构的变更**（改流程、加字段、改约束……）→ **直接更新对应 spec**（保持「系统参考」准确），并按 §4.5 在验收门附 diff。
- 对话中冒出**新需求 / bug / 待办**（哪怕一句话带过）→ **直接加进任务清单**：写入 `docs/tasks.json` 对应批次（来源 `cli`、当天没有就新建一个 cli 批次），每条带 `specRef`（能对上就填）。
- 实现/修完某条 → **回写该 task=done**；涉及 spec 的同时更 spec。
- 拿不准算变更还是闲聊 → **简短问一句**再落；别擅自大改 spec，但**别让需求/任务无声消失**。
> 一句话：**人在 CLI 随口说的事，最终要么进 spec、要么进任务清单**——你负责当场归位，而不是等人去点 `/intake` 或 `/spec`。

## 1. 方法论 TL;DR
- **主线是任务**：用户看的是任务清单（批次/task）；**spec 是系统参考资料**，在进件时被动维护，每条 task 回链它落地的 spec。
- **spec 为系统事实源**：要理解系统全貌/规约看 `docs/specs/*`；改系统行为先把对应 spec 改对（先改 spec → 改测试 → 改码）。
- **统一进件流程**：先析 spec（大小分流）→ 再出任务批次 → 人确认（见 §0.5）。
- **文件即接口**：靠 `docs/tasks.json`（任务） / spec / 代码 / 测试 / 交付说明交接，不靠聊天——可追溯、可重放。
- **小步**：一条 task ≈ 一个 PR ≈ 一次验收（它落地的 spec 随之同步）。
- **不全凭自证**：靠开发自检 + **连真库冒烟** + 人验收兜底（见 §3/§4）。

## 2. 触发指令（自然语言或斜杠命令均可）
| 你说 | 斜杠命令 | 编排器动作 |
|---|---|---|
| 进件（飞书文档/窗口直接录入一堆 bug/需求） | `/intake <链接或文字>` | **统一流程**：拉正文看图 → 先析 spec（大小分流）→ 拆成 N 条 task 落成一个批次 → 人确认 spec diff + 任务拆分（§0.5/§4.6） |
| 手动加一条小任务 | `/todo <事项>` | 作为 task 追加进「手动来源·当天」批次（不再有独立速记） |
| 看/整理任务清单 | `/tasks [批次/状态]` | 按批次列 task、改状态、回链 spec |
| 绿地启动（只有 PRD+原型、没代码） | `/init` | 定栈 + 填①本项目信息 + 搭最小工程骨架 + .gitignore → 再 `/spec` |
| 扫码建 spec（导入存量项目） | `/scan [模块]` | 按模块逆向抽取 `docs/specs/<ID>.md` 草稿（便宜模型·增量）→ 待评审 |
| 把 <模块/需求> 转成 spec | `/spec <x>` | 产出/更新 spec（status=draft）→ 列要点给你评审 |
| 实现 / 开发 <spec-id 或 task> | `/build <id>` | 走 §3 构建流水线；做完回写对应 task=done |
| 修复 <缺陷/task> | `/fix <x>` | 定位 → 析 spec/测试 → 改码 → 回归 → 出报告 → 回写 task |
| 验收 <spec-id> | `/accept <id>` | 汇总验收报告 + 人工验收清单（含 spec diff 确认） |

> **日常入口 = `/intake`**：任何一堆 bug/需求进来都从这走（飞书链接或直接贴文字），它先析 spec、再出任务批次、一起请你拍板。单条小事走 `/todo`。`/spec`·`/build`·`/fix` 是流程内的细分步骤——进件确认后，每条 task 据其 `specRef` 走 `/build`（关联 spec）或 `/fix`。

> **绿地启动（新项目）**：产品刚出 PRD+原型、还没代码 → 先 `/init`（定栈 + 搭最小工程骨架），此时 **PRD（`docs/PRD.md`）+ 原型图（`docs/prototype/`）= 唯一事实源**；骨架就绪后 `/spec` 把 PRD 拆成 spec 树 → 评审 draft→ready → `/build`。
> **存量项目导入**：先 `/scan` 把现有代码抽成 spec 基线（草稿，**以代码为准、PRD 仅参考**），你评审 draft→ready 后即作为事实源；之后的新需求/改动走 `/spec`·`/build`·`/fix` 正向流程。
> 两者方向相反：`/init` 从 PRD/原型**正向**搭起，`/scan` 从已有代码**逆向**抽取。

## 3. 构建流水线（精简版 · `/build <spec-id>`）
**一个开发 agent 干完整条，人做验收。** task 来自批次（`docs/tasks.json`），实现时回链它落地的 spec + task。三个节点：`dev`（写/补 spec + 实现）→ `test`（写测试 + **连真库冒烟**，若本项目有真实库）→ `accept`（人验收）。

1. 读 task 关联的 `docs/specs/<id>.md`（`specRef`；没有就让开发先补）；开做时把该 task `status` 置 `doing`。含 `NEEDS-HUMAN` → 在 `docs/.state/<id>.json` 置 `wait` 节点（控制台提醒）并停下等人。
2. `Agent(dev)` 一气做完：**规约（若缺）→ 实现 → 写测试 + 连真库冒烟 → 编译 + 跑测试**。依 spec + 「①本项目信息」里的接口契约 / 界面基线 / **真实库结构文件**。失败自己修，编译/测试不过不交付。
3. 全绿 → `docs/.state/<id>.json` 置 `accept=wait`（控制台 🔔 提醒你验收）。产出交付说明（改动文件、覆盖 AC、**真库冒烟关键返回**、`NEEDS-HUMAN` 项、风险）。
4. **你验收**：通过 → spec `status=accepted`、`accept=pass`，**回写对应 task `status=done`**；有问题 → `/accept` 闭环（开发自动接手修）。

> **人工触点 = 只有 accept 节点。** `wait`=待验收、`fail`=需决策；其余 todo/doing/pass 全自治。
> **必须派真子代理**：`Agent(dev)` 干活，编排器自己不写业务码。
> **进度回写**：每节点开始置 `doing`、完成置 `pass/fail`，每次写 `docs/.state/<id>.json`（如 `{"nodes":{"dev":"doing"}}`→`{"dev":"pass","test":"doing"}`）再跑 `node tools/board.mjs`。不要手改 board.md。
> **并行**：互不相干的模块可并行派多个 `Agent(dev)`；同模块内串行，避免改同一文件冲突。

## 4. 护栏（硬规则）
- **🚫 禁止臆造库表列名/表名**：写实体 / Mapper / SQL / 任何持久层代码前，**先对照「①本项目信息 → 真实库结构文件」逐字段核对**（遗留命名常与直觉不符，如某列叫 `user_code` 而非 `account`——以真实结构为准）。该文件标「无」、或找不到对应表/列 → 标 `NEEDS-HUMAN` 停下，**绝不假设**。
- **测试不能只 mock**：本项目若有真实库，每个 spec 至少补一条**连真库**的冒烟用例（或在交付说明注明已对真实库结构核过字段映射）——只 mock 抓不到"代码列名 vs 真实库"的错配。
- 复用现有表、不擅改库；接口严格按「接口契约」；命名/分层按「架构/命名规范」。
- 评审必查（依「数据权限/留痕规则来源」）：数据权限、按钮级权限、操作留痕、事务、幂等、并发锁、安全。
- 界面以「界面基线」为唯一基线（若有）。

## 4.5 spec 同步纪律（spec=系统参考资料，进件/改动时被动维护，别过期）
> spec 不再是日常操作对象，但**必须随改动保持准确**（它是 AI/人理解系统的事实源）。进件（`/intake`）第一步就是「先析 spec」（§0.5）；开发改动后同样按下面分类。**核心：先分类并起草 spec diff，只有"涉及 spec"的改动才请人确认，且确认并入验收门——不每次改动都弹，避免疲劳。**

**1) 开发 agent 改完先分类（三类 + 一例外）：**
- **逻辑/行为调整**（改规则、加字段、改流程、新需求）→ **要改 spec**：直接起草 spec 的 diff（改哪条 AC / 新增哪节）。
- **纯 bug 修复**（代码本就该这样、spec 是对的）→ **不改 spec**，只在 `docs/changes/CHG-*.md` 记一条。
- **重构/改名/性能**（行为不变）→ **不改 spec**，CHG 记一条。
- *例外*：bug 暴露了 spec 含糊/漏边界 → **往 spec 补一条回归 AC**（把被坑的场景写清楚），算"涉及 spec"。

**2) 只对"涉及 spec"的改动请人确认，并入验收门：**
- 涉及 spec → 在交付/验收材料里**附上 spec 的 diff**（不是 yes/no，是可审的具体改动）；进件在 `/intake` 第 4 步、开发在 `/accept` 时连同 diff 一起请人拍板；通过则合并 diff + 置 `accepted`，打回则连 spec 一起返工。
- 不涉及 spec（bug/重构）→ 不打扰，自动留 CHG 痕迹即可。

**3) 防漂移**：选"无需改 spec"时，CHG 里记一行原因留痕；定期可让开发对一遍"spec vs 代码"，揪出没同步的。

**4) task 与 spec 的回链**：每条 task 在批次里带 `specRef`（落地到哪条 spec，没有就空）；改动涉及的 spec 即该 task 的 `specRef`。task 做完回写 `status=done`（§3）。

## 4.6 多需求拆解 + 影响面分析（动手前必做，`/intake`·`/spec`·`/fix` 都走这一步）
> 一次进件/提交可能是**多条**需求/改动，且改一处常**波及多处**。规矩：**先拆、先算波及面、让人确认，再实现**——别囫囵当一条做、别只改一处不管牵连。拆出的 N 条同时是**任务批次里的 N 条 task**（§0.5），各自带它落地的 `specRef`。

**1) 拆需求**：把这次进件/提交**拆成 N 条独立需求/改动**列出（每条一句话 + 归属模块），即批次里的 N 条 task。多条要额外看它们**彼此之间**有没有冲突/重叠。

**2) 影响面分析**（每条 + 跨条）：实现前，顺这几条线索扫出**所有受影响的已有 spec**：
- `depends_on` 依赖图（谁依赖目标 spec）
- 共享**数据表 / 字段**（动了同一张表/列）
- 共享**接口**（同一 API 被多模块调用）
- 同**模块**内相邻功能
- **多需求之间**：是否改同一文件/表/接口/AC（标出冲突点）

**3) 出「影响面清单」给人确认**（与 spec 评审同为人工关卡）：列出 每条需求/task → 要新增/改哪些 spec 与 AC（即该 task 的 `specRef`，大特性→新建 draft、小改→改某 AC、不涉结构→空）、波及哪些模块、原因；以及多需求间的交叉点 + 建议处理顺序。进件场景下这一步即 `/intake` 的「spec diff + 任务拆分」确认门。

**4) 一次性处理**：确认后，**所有受影响 spec 一并更新**，按依赖顺序、避免冲突地实现（改同一文件串行、跨模块并行），统一交付、统一验收——不要一条条孤立做、互相覆盖。

## 4.7 经验库 / 防回归（解决过的问题，别再犯第二次）
> **两层"踩坑记忆"**，适用本项目所有会话（不只 `/build`·`/fix`，**直接在终端自由对话解决问题时也一样**）：
> - **项目级** `docs/lessons.md`：本项目专属的坑——特定样式/主题色、本项目字段或模块怪癖、这套栈的专属陷阱。随本项目提交。
> - **全局级** `$STEWARD_LESSONS`（steward 仓库 `lessons.md`，提交进 git、**跨项目跨人共享**）：通用的坑——分页/搜索/筛选交互是否生效统一、表单校验通用陷阱、某框架库的通用坑……换个项目也会犯的。

1. **会话开始：两层都读** —— `cat docs/lessons.md` + `cat "$STEWARD_LESSONS"`，扫各自「✅ 自检清单」+ 相关教训，避开已知坑。
2. **解决非显然的坑后立即回写，并分流**：只对本项目成立 → 写 `docs/lessons.md`；换个项目也会犯（通用）→ 写 `$STEWARD_LESSONS`。把可复用检查点提炼进对应文件的「✅ 自检清单」。显然的小改不用记。
3. **想手动记**：随时 `/lesson <现象/解法>`，它会自动判断该进项目级还是全局级（最可靠，不靠模型自觉）。
4. **人只在需要时扫一眼**新增教训对不对——错的删、好的留，不打扰日常。

> ⚠️ 自由对话里"自动回写"是尽力而为（模型不一定每次都记得）；**要确保记下，就用 `/lesson`**。
> 与 §4.5 的关系：§4.5 管「spec 要不要同步」（行为/规则层）；§4.7 管「踩坑经验沉淀」（实现/排查层，常是 spec 不覆盖的样式/交互/环境坑）。两者互补。
