> **目的：解决过一次的通用问题，别在任何项目再犯。** 这是 **Steward 工具自身沉淀的"踩坑记忆"**——随 steward 仓库走、提交进 git，**跨所有项目、跨所有使用者共享**。
> **只记通用的坑**：换个项目也会犯的那种——分页/搜索/筛选交互是否生效统一、表单校验通用陷阱、某框架/库的通用坑、跨项目的接口/分页约定……

> 文件位置：steward 仓库根的 `lessons.md`（环境变量 `$STEWARD_LESSONS` 指向它）。`cat "$STEWARD_LESSONS"` 读、Edit 写回。

## 规矩（dev agent 必须遵守）
1. **开工前**：先扫一遍下面的「✅ 自检清单」+ 与本次改动相关的「📚 教训明细」，避开已知坑。
2. **收尾时**：本次若踩了一个**非显然**的坑（查了一会儿才搞定、或很可能再犯的那种），**追加一条教训**（格式见文末模板），并把其中可复用的检查点**提炼进「✅ 自检清单」**。
3. **判断要不要记**：显然的小改不用记；值得记的是「下次别人/AI 很可能再踩同一个坑」。
4. 验收/复验时，人会顺带看一眼新增的教训对不对——错的就删，好的留。

## ✅ 自检清单（每次交付前逐条过）
> 下面是从历次教训提炼出的可复用检查点，是本文件最有价值的部分。开工与收尾各过一遍。

- [ ] 改了列表/表格/卡片样式 → 长文本、空数据、超多列/超多行都看过，没溢出/没错位
- [ ] 加了筛选/搜索条件 → 确认条件**真传到后端并生效**（不是只改了 UI、后端没接）
- [ ] 写持久层 → 列名/表名**逐字核对真实库结构**，没臆造（遗留命名常反直觉）
- [ ] 改了某处共享逻辑 → 按影响面把**所有调用方/受影响 spec** 一并过了
- [ ] 列表/表格页布局先数搜索条件：**少（≤2~3 个）就单行**——查询/重置按钮紧跟搜索框后面，别另起一行
- [ ] **筛选项只有 1 个（简单文本搜索框/单个下拉）→ 不放前置 label**，直接靠 placeholder（label 与 placeholder 重复多余）；**≥2 个同类筛选项才放 label** 便于区分对齐。例外：日期范围筛选始终保留合并 label（如「出院日期」「申请日期」），因范围控件占位不够自解释（见 UI 规范第 9 条）。
- [ ] **派生/聚合数据时，前端分组/筛选依赖的字段必须一并透传**：生成 board.json / 汇总 JSON 这类「派生数据」时，凡是 UI 拿来**分组、筛选、排序**的字段（如 `module`/`tag`/`owner`）都要写进去——漏一个，对应功能会**静默退化**（如"按模块分组"全落到「未标模块」），且常在重新生成/重启后才暴露，排查时容易误判成数据丢了。改派生脚本后，对一遍「源数据有哪些字段 vs 前端实际用到哪些字段」。
- [ ] **接飞书 wiki 文档**：除了 `docx:document:readonly`，还要开 `wiki:wiki:readonly`（解析知识库节点）+ 把该**知识库共享给应用**；只开 docx 会在 `wiki/v2/spaces/get_node` 阶段报 `99991672` 权限不足。按"文档在哪类容器(云空间文件夹 / 知识库 wiki)"决定开哪套权限 + 共享对应容器。
- [ ] **"打开菜单"的可点区域别直接绑破坏性动作**：顶栏用户名/头像、行内"更多"这类应该**点了弹下拉菜单**的区域，整块 `@click` 不要直接绑 `logout`/`delete` 等动作——否则点一下就执行（用户以为只是展开菜单）。正确：用下拉容器（如 `el-dropdown`）包住可点块，破坏性动作放进菜单项、由其 command 触发，并加二次确认。多子应用/多页面同种顶栏要**全量对齐**（一个有 bug 往往全有），可拿已正确的那个当范式照抄（见 [[L-006]] pams 顶栏点名字即登出）。
- [ ] **开关类「即选即存」（el-switch 切换即保存生效）**：影响外发/启停等**有副作用**的总开关，别埋在「保存」按钮里（用户切了常忘点保存、以为生效其实没存）。做法：① el-switch 加 `@change`（**注意 `v-model` 已先把绑定值置成新值、@change 才触发**，故"新值"= `v===true`、"原值"= `!新值`）；② 切换先 `ElMessageBox.confirm` **二次确认**（启用/关闭两个方向各一句明确后果提示）；③ 确认 → 立即调保存接口（payload 用"当前已保存配置 + 新开关值"，敏感字段如 apiKey 传掩码/空=不改）→ 成功提示「已…，即时生效」+ 回读确认；④ **取消 / 保存失败 → 还原开关回原值**（别留「切了但没生效」的视觉假象）。**防循环**：el-switch **程序化赋值（回读回写 / 取消还原）默认不触发 @change**，故无「保存→回读→再触发 @change」死循环；仍可加一个 `switching` 标志位 + 还原函数（守卫期间 @change 直接 return）作纵深防御。另一等价写法是改用单向 `:model-value` + `@change`（不用 v-model），从根上不回写、无循环可能（pams datacenter「文件合并规则」启停列即此法）。
- [ ] **登录前要读的「公开配置/品牌」必须单独开免鉴权端点 + 精确白名单，别复用要鉴权的配置接口**：登录页（登录前，无 token）要展示医院 Logo/单位名/主题色等系统配置时，通用配置接口（如 `/config/list`）是**要鉴权的**——直接调会 401、页面空白。做法：① 后端**新开一个只读公开端点**（如 `/auth/brand`），**只返回展示必需的少数字段**（logo/company/name），**绝不透传整份配置**（防敏感项泄露）；② 在鉴权白名单**精确放行该路径**（与 `/login` 并列，别用 `/**` 放太宽）；③ 前端用**原生 fetch 直连**该端点、**绕过**带 Authorization 头 + 失败弹 toast 的 axios 拦截器，任何失败（网络/超时/非 200）**静默回退写死文案**、绝不阻塞登录；④ 配置缺失/JSON 非法时端点返回**空字段**（不报错），前端据空回退。附带图片资源（如 Logo 走 `/file/...`）确认其**由静态服务/nginx 在同源根路径提供、不经业务鉴权拦截器**，则登录前天然可加载、无需额外放行（`/file/**` 不在 `/api/**` 拦截范围内）。多个子应用登录页共用**同一套取数 + 回退**工具，避免各写各样（见 pams UC-01 AC-9 四登录页读 `sys_config.hospital_config`）。
- [ ] **状态徽章/标签别用「文案」判色，按「状态码/枚举」上色**：用文案正则匹配着色（如 `/已?归档/→绿`）会把语义相反的态误判——「**未归档**」被 `已?归档` 命中判绿、与「已归档」同色，肉眼无法区分。凡有稳定状态码（后端 enum / DB 枚举列）的字段，前端建**独立的「码→{中文,颜色}」映射**（逐字段镜像后端枚举，颜色/中文对齐、别抄错），徽章按**状态码**取色、下拉筛选也用同一份映射生成全量选项；未知/空态兜底中性色 + 显示原值、不报错。**别改通用文案判色函数**（其它页在用，避免回归），新建映射自用。（来源：pams BMS-02 病案状态列，`emr_status` 9 态。）

# 全局工作规范（所有项目共用）

> 本文件被每个项目的 Claude Code 会话自动加载，是**跨项目通用约定**。
> 项目级 `CLAUDE.md` 与此冲突时以项目级为准；此处只放与具体项目无关、可复用的规范。

## 前端 UI 通用规范

适用于所有前端项目的**列表页 / 后台管理界面**。新页面默认照此实现，已有页面逐步对齐。
（来源：病案平台 pams 的用户反馈，沉淀为通用规范。）

1. **菜单分级分组**：侧边/导航菜单按业务域**分组分级**（父分组 + 子菜单，可展开/折叠），不要一长串平铺罗列。

2. **菜单稳定排序**：菜单项按固定 `sort` 排序，顺序**稳定且符合业务优先级**；不要依赖插入顺序或随机顺序。

3. **表格内部滚动，禁止整页滚动**：列表页表格区域**自身内部滚动** + 表头 sticky 固定；页面外壳固定视口高度，**不出现 body 全局滚动条**。
   骨架：外壳 `height:100vh; box-sizing:border-box; overflow:hidden; display:flex; flex-direction:column`；工具条/分页 `flex:0 0 auto`；表格外层 `flex:1; min-height:0; overflow:auto`；`thead th { position:sticky; top:0 }`。

4. **搜索/筛选条件可收缩**：表头筛选区**超过一行**时做成可展开/收起——默认只显示第一行 + "展开/收起更多"，避免 2+ 行筛选挤占垂直空间、压缩表格可视内容。

5. **内容卡片紧凑**：主内容卡片 **padding 收紧**，减少大块留白与间隙，保持紧凑美观（避免默认大 padding 把表格挤到很小一块）。

6. **造数验证分页**：每个功能模块交付时**造一批足够触发多页的样本数据**，端到端验证分页（翻页、每页条数、总数统计）正确，而不是只有几条看不出分页问题。

7. **统一"返回门户"入口**：多子应用 + 门户的架构下，**每个子应用都要有统一的"返回门户"入口**（prod 跳同源门户 `/`、dev 跳门户开发端口）；不要某些子应用有、某些没有。

8. **列表搜索条件持久化**：列表页的搜索/筛选条件 + 分页（页码、每页数）要**记录下来**——从列表点进详情、再返回列表时，**原筛选条件与所在页要保留**，不能被重置成默认。实现可用路由 query 同步、`keep-alive` 缓存列表组件、或状态store；返回路径要能命中缓存。

9. **起止日期用单个范围框**：开始时间/结束时间这类成对条件，做成**一个范围控件**（一个单元格内"起 ~ 止"），**不要拆成两个独立筛选格**——否则栅格换行时会把一对拆到两行，很别扭。label 用合并名（如 申请起/申请止 → "申请日期"）。

10. **栅格筛选列数统一**：同一系统所有列表页的筛选栅格**列数一致**（如统一一行 4 个），列宽对齐；折叠时也保持满行（首行就是完整一行 N 个，不要收起后只剩 N-1 个 + 空格）。"展开/收起"只在筛选项确实超过一行时才出现；不足一行的页面不显示折叠按钮。

11. **选择类筛选即选即查**：**下拉/单选/多选/日期(范围)选择器/可搜索下拉**等"选择型"筛选项，**选中即触发查询**（`@change` → 重置到第 1 页 + 重新请求接口），**不用再点"查询/搜索"按钮**。只有**自由文本输入框**才靠回车或点搜索按钮触发（避免每敲一个字就请求）。所有列表页、所有子系统统一此交互。

12. **时间统一格式 `yyyy-MM-dd HH:mm`**：所有**展示**页面（列表列、详情、抽屉、卡片…）的日期时间一律格式化成 `yyyy-MM-dd HH:mm`（年-月-日 时:分，24 小时制），**不要直接显示后端原始值**（如 ISO 的 `2025-04-09T18:58`、带毫秒/时区、时间戳）。用一个**共享格式化函数**统一处理（兼容 ISO 串 / 时间戳 / Date），空值显示占位（如 `—`）。纯日期字段无时间部分时可只到 `yyyy-MM-dd`，但凡有时间的统一到分钟。

13. **表单按真实库字段长度做校验（前后端双层 + 好的字段提示）**：长度等约束**对齐真实数据库列定义**（如 `varchar(20)` → 最多 20）——**先查真实表结构（如 `docs/db-schema.sql`）逐字段核对，禁止臆造**，查不到对应列就停下问人。
    - **前端**：输入项加 `maxlength` + 提交前校验；并**展示好字段提示**——让用户**提前看到约束**（如字数计数 `12/20` / `≤20字` 的 helper）、**校验失败精确定位到出错字段**（内联红字提示，别只弹一个笼统 toast）。
    - **后端**：save/update DTO 字段补 `@Size(max=N)`/`@NotBlank`/格式校验（与前端同一套长度），做**纵深防御**——绕过前端直调接口也不会超长入库报错；校验信息要能被前端清晰展示。
    - 数字范围、必填、格式（手机号/证件号等）同理按库字段约束前后端都校验。

## 📚 教训明细
> dev agent 把新教训追加在这里。

### L-001 搜索条件少的表格页，筛选按钮另起一行浪费纵向空间、视觉割裂
- 范围/模块：通用 · 前端列表/表格页布局
- 根因：默认把筛选条件区和「查询/重置」按钮分两行排，字段少时显得空、割裂。
- 解法：搜索条件少（≤2~3 个）时，把按钮**直接紧跟在搜索框后面、同一行**，不另起一行。
- 防复发：列表页布局先数搜索条件数量——少就单行（按钮跟随）；多（或带高级筛选）才换多行/可折叠。
- 关联：UI 规范，无 spec。

### L-002 单个简单筛选项还放前置 label，与 placeholder 重复多余
- 范围/模块：通用 · 前端列表/表格页筛选区
- 根因：每个筛选项习惯性配一个 `<label>` + 输入框，但当全页只有 1 个简单搜索框（或 1 个下拉）时，label 文字和输入框 placeholder 内容基本重复（如 label「箱编码/箱名」+ placeholder「箱编码 / 箱名」），白占空间、看着啰嗦。
- 解法：**筛选项只有 1 个简单文本搜索框 / 单个下拉 → 去掉前置 label，仅留输入框（placeholder 已说明）**；只有当**同类筛选项有 2 个及以上**、需要逐个区分对齐时才放 label。例外：日期范围筛选始终保留合并 label（范围控件占位不自解释，见 UI 规范第 9 条）。
- 防复发：见自检清单对应项；新列表页先数筛选项数量决定要不要 label。
- 关联：[[L-001]]（同属"按筛选项数量决定布局"）；UI 规范第 9/10 条。

### L-003 派生脚本漏传分组字段，前端「按模块分组」静默退化、重启后才暴露
- 范围/模块：通用 · 派生/聚合数据脚本 ↔ 前端分组/筛选
- 现象：spec 文件头都写了 `module:`，前端也有「按模块分组」，但实际全归到「未标模块」；门户/数据中心等命名分组消失，且是服务重启重新生成 board.json 后才发现。
- 根因：`board.mjs` 把 spec frontmatter 派生成 `board.json` 时，`specs.push({...})` 只挑了 id/title/priority/status，**漏了 `module`**。前端按 `sp.module` 分组，拿到 undefined 就全落到「未标模块」——功能没报错、只是静默退化。
- 解法：派生 `board.json` 的 `specs.push` 补上 `module: (fm.module||'').trim()`；并把 5 个存量项目的 board.mjs 一并同步、重算看板。
- 防复发：见自检清单「派生数据时前端依赖字段必须一并透传」；改任何"源→派生数据"的脚本，先列清楚前端到底用了哪些字段，逐一核对别漏。
- 关联：UI 规范无；steward 控制台看板分组。

### L-004 飞书 wiki 文档读不到：光开 docx 权限不够，wiki 要单独开权限 + 共享知识库
- 范围/模块：通用 · 飞书开放平台文档接入（steward `/intake` / feishu-fetch）
- 现象：配好飞书自建应用、开了 `docx:document:readonly`，拉 `/wiki/` 链接报 `99991672 Access denied. scopes required: wiki:wiki / wiki:wiki:readonly / wiki:node:read`。
- 根因：知识库(wiki) 与云文档(docx) 是**两套权限**。`/wiki/<token>` 要先调 `wiki/v2/spaces/get_node` 解析节点（需 wiki 读权限），解析出的 docx 再读正文（才用 docx 权限）。只开 docx → 卡在 get_node。
- 解法：开放平台给应用加 `wiki:wiki:readonly`（+`docx:document:readonly`）→ **重新发布版本**（权限变更要重发才生效）→ 并把**该知识库共享给应用**（光有 scope、不共享具体知识库也读不到）。
- 防复发：见自检清单对应项；飞书接入先看"文档在哪类容器"再决定权限 + 共享。
- 关联：steward 飞书接入（feishu-fetch / `/intake`）；无 spec。

### L-005 飞书 raw_content 丢图片，需求藏在截图里 → 漏需求
- 范围/模块：通用 · 飞书文档接入（steward `/intake` / feishu-fetch）
- 现象：`/intake` 拉飞书需求文档后分诊、做出来验收，**第一项就没完成**——因为那条需求是用**截图**说明的，而 `docx/v1/.../raw_content` 只回纯文本，图片全丢（顶多剩个 `xxx.png` 占位）。
- 根因：飞书文档接口 raw_content 不含图片；需求文档又极爱用截图标注界面改动。只读文本 = 漏掉一半需求。
- 解法：feishu-fetch 额外用 `docx/v1/documents/{id}/blocks` 按序收集 image block(block_type 27) 的 token → `drive/v1/medias/{token}/download` 下载到 `docs/.intake/<docId>/` → 在输出末尾列出路径；`/intake` 让 claude **逐张 Read（视觉）** 并入分诊。下载需应用额外开 **`drive:drive:readonly`** 权限（光 docx 权限会 99991672）。
- 防复发：飞书接入要把"图片也取下来给模型看"当默认；缺 drive 权限时 fetcher 明确报"图片没拉到、先补权限别硬分诊"。
- 关联：[[L-004]]（飞书权限分容器）；feishu-fetch / `/intake`。

### L-006 顶栏用户名/头像整块绑了登出，点名字就直接退出登录
- 范围/模块：通用 · 前端顶栏用户菜单（多 SPA 子应用壳层 Layout）
- 现象：进子系统后点右上角「用户姓名」直接退出登录跳回登录页，预期应弹账户下拉菜单（修改密码/退出登录），登出只在点「退出登录」时发生。pams 三个子应用（BMS/用户中心/数据中心）全中。
- 根因：壳层 `Layout.vue` 把整个用户名区域 `.tb-u` 直接 `@click="logout"`（唯一可点项就是登出），点名字即执行。门户 Portal.vue 用 `el-dropdown` 包住才是对的——同种顶栏多处复制，一个写错往往全错。
- 解法：用下拉容器（`el-dropdown trigger="click" @command`）包住 `.tb-u`，破坏性动作放进菜单项、由 command 触发 + 二次确认；以项目里已正确的那个（门户）为范式照抄，多子应用全量对齐。子应用没有改密弹窗就只放「退出登录」。
- 防复发：见自检清单「打开菜单的可点区域别直接绑破坏性动作」；审顶栏/行操作时，凡"应该弹菜单"的可点块都核一遍它的 @click 是不是误绑了动作。
- 关联：pams CHG-2026-06-24-topbar-username-direct-logout。

### L-007 登录页要读的公开配置/品牌，复用了要鉴权的配置接口 → 登录前 401 拿不到
- 范围/模块：通用 · 登录页（登录前，无 token）读系统展示配置（Logo/单位名/主题色/品牌）；多 SPA 子应用各自登录页
- 现象：需求「4 个子系统登录页统一读医院配置展示品牌」。医院配置在 `sys_config.hospital_config`（JSON：logo/company/hospitalName），但既有读取接口 `/usercenter/config/list` **要鉴权**，而登录页在登录前、没 token；直接调会 401，登录页拿不到品牌、只能停在写死文案。
- 根因：把「登录后才用的、要鉴权的配置接口」当成「登录前也能用」的来源。鉴权拦截器只放行了 `/auth/login`，公开配置端点不存在。
- 解法：① 后端新开**免鉴权只读端点** `GET /auth/brand`，读 `hospital_config` JSON 但**只返 logo/company/hospitalName 三项**（不透传整份配置，防敏感项外泄）；缺失/空/JSON 非法 → 返回空串字段、不报错。② 白名单**精确放行** `/api/usercenter/auth/brand`（抽成 `WebMvcConfig.PUBLIC_PATHS` 常量、与 `/auth/login` 并列，并写自检测试断言含该路径）。③ Logo 图 `/file/SYS_LOGO/**` 由 nginx 同源根路径静态服务、**不经 `/api/usercenter/**` 拦截器**，登录前天然可加载，无需额外白名单。④ 前端 4 个登录页共用 `utils/brand.ts` `loadBrand()`：**原生 fetch** 直连，绕过带鉴权头/失败弹 toast 的 axios 拦截器；超时(AbortController)/网络失败/非 200 一律**静默回退**各页写死文案与默认图标，绝不阻塞登录。
- 防复发：见自检清单「登录前要读的公开配置/品牌必须单独开免鉴权端点 + 精确白名单」。凡「登录前/未登录页面要读后端数据」，先问：这条链路要不要 token？要 → 单开公开端点 + 精确放行 + 只返必需字段 + 失败静默回退。
- 关联：pams UC-01 AC-9 / UC-06（hospital_config 品牌公开消费）；[[L-037]]（sys_config 一行 JSON 存法，键名先查真库别臆造）。
