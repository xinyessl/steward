# 全局工作规范（所有项目共用）

> 本文件被每个项目的 Claude Code 会话自动加载，是**跨项目通用约定**。
> 项目级 `CLAUDE.md` 与此冲突时以项目级为准；此处只放与具体项目无关、可复用的规范。

## 前端 UI 通用规范

适用于所有前端项目的**列表页 / 后台管理界面**。新页面默认照此实现，已有页面逐步对齐。
（来源：病案平台 pams 的用户反馈，沉淀为通用规范。）

1. **菜单分级分组**：侧边/导航菜单按业务域**分组分级**（父分组 + 子菜单，可展开/折叠），不要一长串平铺罗列。

2. **菜单稳定排序**：菜单项按固定 `sort` 排序，顺序**稳定且符合业务优先级**；不要依赖插入顺序或随机顺序。

3. **表格内部滚动，禁止整页滚动**：列表页表格区域**自身内部滚动** + 表头 sticky 固定；页面外壳固定视口高度，**不出现 body 全局滚动条**。
   骨架：外壳 `height:100vh; box-sizing:border-box; overflow:hidden; display:flex; flex-direction:column`；工具条/分页 `flex:0 0 auto`；表格外层 `flex:1; min-height:0; overflow:auto`；`thead th { position:sticky; top:0 }`。

4. **搜索/筛选条件可收缩**：表头筛选区**超过一行**时做成可展开/收起——默认只显示第一行 + "展开/收起更多"，避免 2+ 行筛选挤占垂直空间、压缩表格可视内容。

5. **内容卡片紧凑**：主内容卡片 **padding 收紧**，减少大块留白与间隙，保持紧凑美观（避免默认大 padding 把表格挤到很小一块）。

6. **造数验证分页**：每个功能模块交付时**造一批足够触发多页的样本数据**，端到端验证分页（翻页、每页条数、总数统计）正确，而不是只有几条看不出分页问题。

7. **统一"返回门户"入口**：多子应用 + 门户的架构下，**每个子应用都要有统一的"返回门户"入口**（prod 跳同源门户 `/`、dev 跳门户开发端口）；不要某些子应用有、某些没有。

8. **列表搜索条件持久化**：列表页的搜索/筛选条件 + 分页（页码、每页数）要**记录下来**——从列表点进详情、再返回列表时，**原筛选条件与所在页要保留**，不能被重置成默认。实现可用路由 query 同步、`keep-alive` 缓存列表组件、或状态store；返回路径要能命中缓存。

9. **起止日期用单个范围框**：开始时间/结束时间这类成对条件，做成**一个范围控件**（一个单元格内"起 ~ 止"），**不要拆成两个独立筛选格**——否则栅格换行时会把一对拆到两行，很别扭。label 用合并名（如 申请起/申请止 → "申请日期"）。

10. **栅格筛选列数统一**：同一系统所有列表页的筛选栅格**列数一致**（如统一一行 4 个），列宽对齐；折叠时也保持满行（首行就是完整一行 N 个，不要收起后只剩 N-1 个 + 空格）。"展开/收起"只在筛选项确实超过一行时才出现；不足一行的页面不显示折叠按钮。

11. **选择类筛选即选即查**：**下拉/单选/多选/日期(范围)选择器/可搜索下拉**等"选择型"筛选项，**选中即触发查询**（`@change` → 重置到第 1 页 + 重新请求接口），**不用再点"查询/搜索"按钮**。只有**自由文本输入框**才靠回车或点搜索按钮触发（避免每敲一个字就请求）。所有列表页、所有子系统统一此交互。

12. **时间统一格式 `yyyy-MM-dd HH:mm`**：所有**展示**页面（列表列、详情、抽屉、卡片…）的日期时间一律格式化成 `yyyy-MM-dd HH:mm`（年-月-日 时:分，24 小时制），**不要直接显示后端原始值**（如 ISO 的 `2025-04-09T18:58`、带毫秒/时区、时间戳）。用一个**共享格式化函数**统一处理（兼容 ISO 串 / 时间戳 / Date），空值显示占位（如 `—`）。纯日期字段无时间部分时可只到 `yyyy-MM-dd`，但凡有时间的统一到分钟。

13. **表单按真实库字段长度做校验（前后端双层 + 好的字段提示）**：长度等约束**对齐真实数据库列定义**（如 `varchar(20)` → 最多 20）——**先查真实表结构（如 `docs/db-schema.sql`）逐字段核对，禁止臆造**，查不到对应列就停下问人。
    - **前端**：输入项加 `maxlength` + 提交前校验；并**展示好字段提示**——让用户**提前看到约束**（如字数计数 `12/20` / `≤20字` 的 helper）、**校验失败精确定位到出错字段**（内联红字提示，别只弹一个笼统 toast）。
    - **后端**：save/update DTO 字段补 `@Size(max=N)`/`@NotBlank`/格式校验（与前端同一套长度），做**纵深防御**——绕过前端直调接口也不会超长入库报错；校验信息要能被前端清晰展示。
    - 数字范围、必填、格式（手机号/证件号等）同理按库字段约束前后端都校验。

## 📚 教训明细
> dev agent 把新教训追加在这里。

### L-001 搜索条件少的表格页，筛选按钮另起一行浪费纵向空间、视觉割裂
- 范围/模块：通用 · 前端列表/表格页布局
- 根因：默认把筛选条件区和「查询/重置」按钮分两行排，字段少时显得空、割裂。
- 解法：搜索条件少（≤2~3 个）时，把按钮**直接紧跟在搜索框后面、同一行**，不另起一行。
- 防复发：列表页布局先数搜索条件数量——少就单行（按钮跟随）；多（或带高级筛选）才换多行/可折叠。
- 关联：UI 规范，无 spec。

### L-002 单个简单筛选项还放前置 label，与 placeholder 重复多余
- 范围/模块：通用 · 前端列表/表格页筛选区
- 根因：每个筛选项习惯性配一个 `<label>` + 输入框，但当全页只有 1 个简单搜索框（或 1 个下拉）时，label 文字和输入框 placeholder 内容基本重复（如 label「箱编码/箱名」+ placeholder「箱编码 / 箱名」），白占空间、看着啰嗦。
- 解法：**筛选项只有 1 个简单文本搜索框 / 单个下拉 → 去掉前置 label，仅留输入框（placeholder 已说明）**；只有当**同类筛选项有 2 个及以上**、需要逐个区分对齐时才放 label。例外：日期范围筛选始终保留合并 label（范围控件占位不自解释，见 UI 规范第 9 条）。
- 防复发：见自检清单对应项；新列表页先数筛选项数量决定要不要 label。
- 关联：[[L-001]]（同属"按筛选项数量决定布局"）；UI 规范第 9/10 条。

### L-003 派生脚本漏传分组字段，前端「按模块分组」静默退化、重启后才暴露
- 范围/模块：通用 · 派生/聚合数据脚本 ↔ 前端分组/筛选
- 现象：spec 文件头都写了 `module:`，前端也有「按模块分组」，但实际全归到「未标模块」；门户/数据中心等命名分组消失，且是服务重启重新生成 board.json 后才发现。
- 根因：`board.mjs` 把 spec frontmatter 派生成 `board.json` 时，`specs.push({...})` 只挑了 id/title/priority/status，**漏了 `module`**。前端按 `sp.module` 分组，拿到 undefined 就全落到「未标模块」——功能没报错、只是静默退化。
- 解法：派生 `board.json` 的 `specs.push` 补上 `module: (fm.module||'').trim()`；并把 5 个存量项目的 board.mjs 一并同步、重算看板。
- 防复发：见自检清单「派生数据时前端依赖字段必须一并透传」；改任何"源→派生数据"的脚本，先列清楚前端到底用了哪些字段，逐一核对别漏。
- 关联：UI 规范无；steward 控制台看板分组。

### L-004 飞书 wiki 文档读不到：光开 docx 权限不够，wiki 要单独开权限 + 共享知识库
- 范围/模块：通用 · 飞书开放平台文档接入（steward `/intake` / feishu-fetch）
- 现象：配好飞书自建应用、开了 `docx:document:readonly`，拉 `/wiki/` 链接报 `99991672 Access denied. scopes required: wiki:wiki / wiki:wiki:readonly / wiki:node:read`。
- 根因：知识库(wiki) 与云文档(docx) 是**两套权限**。`/wiki/<token>` 要先调 `wiki/v2/spaces/get_node` 解析节点（需 wiki 读权限），解析出的 docx 再读正文（才用 docx 权限）。只开 docx → 卡在 get_node。
- 解法：开放平台给应用加 `wiki:wiki:readonly`（+`docx:document:readonly`）→ **重新发布版本**（权限变更要重发才生效）→ 并把**该知识库共享给应用**（光有 scope、不共享具体知识库也读不到）。
- 防复发：见自检清单对应项；飞书接入先看"文档在哪类容器"再决定权限 + 共享。
- 关联：steward 飞书接入（feishu-fetch / `/intake`）；无 spec。

### L-005 飞书 raw_content 丢图片，需求藏在截图里 → 漏需求
- 范围/模块：通用 · 飞书文档接入（steward `/intake` / feishu-fetch）
- 现象：`/intake` 拉飞书需求文档后分诊、做出来验收，**第一项就没完成**——因为那条需求是用**截图**说明的，而 `docx/v1/.../raw_content` 只回纯文本，图片全丢（顶多剩个 `xxx.png` 占位）。
- 根因：飞书文档接口 raw_content 不含图片；需求文档又极爱用截图标注界面改动。只读文本 = 漏掉一半需求。
- 解法：feishu-fetch 额外用 `docx/v1/documents/{id}/blocks` 按序收集 image block(block_type 27) 的 token → `drive/v1/medias/{token}/download` 下载到 `docs/.intake/<docId>/` → 在输出末尾列出路径；`/intake` 让 claude **逐张 Read（视觉）** 并入分诊。下载需应用额外开 **`drive:drive:readonly`** 权限（光 docx 权限会 99991672）。
- 防复发：飞书接入要把"图片也取下来给模型看"当默认；缺 drive 权限时 fetcher 明确报"图片没拉到、先补权限别硬分诊"。
- 关联：[[L-004]]（飞书权限分容器）；feishu-fetch / `/intake`。
